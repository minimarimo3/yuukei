# 00 プロダクトコンセプト

## 1. この文書の目的

この文書は、Yuukei を「何のために作るか」「誰のために作るか」「何を実装対象とするか」を定義するための上位文書です。

アーキテクチャや各機能仕様を決める前提として、以下を明文化します。

- プロダクトの理念
- 提供したい体験
- 想定ユーザー
- 機能の優先順位
- やらないこと

## 2. プロダクト概要

Yuukei（有景）は、Unity + VRM で動作する OSS の 3D デスクトップマスコットです。

単なる会話 UI や通知ツールではなく、**ユーザーの PC の中に住んでいるように感じられる存在**を目指します。キャラクターは画面や時刻、ユーザーの操作、ファイルとの接触などをきっかけに振る舞い、デバイス上での生活感を持ちます。

## 3. ビジョン

### 3.1 目指す世界観

Yuukei の中心にあるのは、**「デバイスへの居住」** という考え方です。

キャラクターはアプリの 1 機能として存在するのではなく、PC という空間に住み着いている存在として振る舞います。ユーザーが話しかけたときだけ応答する受け身の AI ではなく、状況に応じて自律的に反応し、ときには前面に出ず静かに過ごします。

### 3.2 価値の置き方

Yuukei が提供したい価値は、タスク効率そのものではなく次の 3 点です。

- キャラクターがそこにいると感じられる実在感
- 日常の PC 操作に寄り添う軽いインタラクション
- ユーザーやコミュニティが振る舞いを育てられる拡張性

## 4. プロダクト原則

### 4.1 会話アプリではなく居住型キャラクター

会話は重要な要素ですが、Yuukei の本質はチャットボットではありません。会話機能は、キャラクターの存在感を補強する手段として扱います。

### 4.2 ユーザーの作業を邪魔しない

キャラクターは常に目立つ必要はありません。表示、発話、演出は「かわいさ」だけでなく、作業中の負荷や邪魔にならなさを考慮して制御されるべきです。

### 4.3 反応は文脈に根ざす

時刻、クリック、ドラッグ、ファイル接触、システム状態など、PC 上の文脈に応じて振る舞うことを重視します。唐突なランダム動作より、理由のある反応を優先します。

### 4.4 振る舞いはユーザーが編集・拡張できる

キャラクターの魅力は固定コンテンツではなく、継続的に育てられることにあります。そのため台本 DSL、プラグイン、パッケージによる拡張はプロダクトの中核機能です。

### 4.5 キャラクター体験と安全性を両立する

ローカル情報や LLM を扱う以上、ユーザーが「何が見られ、何が送信されるか」を把握できる設計が必要です。没入感のために不透明な挙動を許容しません。

## 5. 想定ユーザー

### 5.1 主対象

主な対象は、二次元文化に親和性があり、PC 上でキャラクターと暮らす体験に価値を感じるユーザーです。

具体的には次の層を主対象とします。

- イラストレーター、同人制作者、VTuber モデル制作者などのクリエイター
- 長時間 PC を起動しているゲーマー
- デスクトップの見た目や常駐ソフトにこだわりがあるユーザー
- キャラクターを「便利ツール」より「一緒にいる存在」として楽しみたいユーザー

### 5.2 副対象

- 自作台本やプラグインを書きたい技術志向のユーザー
- 自作キャラクターや配布パッケージを公開したいコミュニティ参加者

### 5.3 想定しない主要ターゲット

次のニーズは Yuukei の中心ターゲットから外れます。

- 業務効率のみを目的とした汎用 AI アシスタント
- テキストチャット中心の高精度対話エージェント
- 企業導入を前提とした厳格な管理ソフト

## 6. 提供したい中核体験

Yuukei は次の体験を実現することを目標にします。

### 6.1 そこに住んでいる感覚

起動していないときも存在を想像でき、起動すると「戻ってきた」と感じられること。単発の会話品質より、継続利用で育つ印象を重視します。

### 6.2 画面上の出来事への自然な反応

クリックされた、特定時刻になった、ファイルの上に重なった、PC が忙しそう、といった出来事に対して、キャラクターが自然に反応すること。

### 6.3 キャラクターごとの差別化

同じエンジン上でも、台本やアセットによって性格・話し方・振る舞いが大きく変わること。Yuukei 自体はキャラクター表現の基盤であるべきです。

### 6.4 自分の手で育てる余地

ユーザーが台本を書き換えたり、パッケージを追加したり、外部機能を導入したりして、体験を育てられること。

## 7. 機能方針

### 7.1 コア機能

Yuukei の中核を成す機能は以下です。

- VRM ベースの 3D キャラクター表示
- 吹き出し、口パク、モーションなどの基本演出
- 台本（Daihon）DSL による振る舞い記述
- トリガー駆動のイベント反応
- 状態保存を伴う継続的なキャラクター体験
- LLM を利用した任意の応答生成
- プラグイン / パッケージによる拡張

### 7.2 初期リリースで優先する機能

初期実装では、まず「居住感のある最低限の体験」を成立させることを優先します。

優先度が高いもの:

- キャラクターの表示、移動、基本アニメーション
- 起動時 / クリック / 時刻 / 定期確認などの基本トリガー
- 台本実行と変数保存
- 吹き出し表示と簡単なユーザー入力
- LLM なしでも成立する振る舞い設計
- キャラクターや台本を差し替えられる基本構造

初期段階では後回しにしてよいもの:

- 高度なエージェント的自律行動
- 複雑なマルチモーダル認識
- 大規模なオンライン共有機能
- モバイル連携やクロスデバイス常駐

### 7.3 LLM の位置づけ

LLM は Yuukei の中心そのものではなく、**振る舞いを補強する拡張手段**として位置づけます。

そのため、以下を原則とします。

- LLM がなくてもキャラクター体験が成立する
- LLM の利用は明示的に設定できる
- クラウド送信を伴う場合はユーザーが理解できる導線を用意する

## 8. 非目標

Yuukei では、少なくとも現時点で次を主要目標にしません。

- 何でもこなす万能デスクトップ AI を作ること
- 業務アシスタントや秘書機能を中心に据えること
- 実在感より効率を優先したミニマル UI に寄せること
- 閉じた専用コンテンツとして完成品だけを提供すること

## 9. 成功条件

Yuukei が狙いどおりのプロダクトになっているかは、次の観点で判断します。

- ユーザーが「便利だから」だけでなく「そこにいてほしいから」起動する
- LLM なしでもキャラクターとして成立している
- キャラクターごとの差分を台本やパッケージで素直に表現できる
- 拡張や改造がしやすく、コミュニティが振る舞いを増やせる
- 常駐していても邪魔、重い、怖いと感じにくい

## 10. 未確定事項

以下は今後の対話で詰めるべき論点です。

- Yuukei を「マスコットソフト」と呼ぶか、「居住型キャラクタープラットフォーム」と呼ぶか
- 初期リリース時点でどこまで LLM 機能を前提にするか
- フォルダや画面内容の把握を、どの範囲まで既定機能として許容するか
- 初期ターゲットを個人利用者に絞るか、創作者コミュニティまで最初から強く意識するか
- 配布単位を「アプリ本体 + キャラパッケージ」としてどこまで明確化するか
